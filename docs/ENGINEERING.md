## 1. Architecture

Two independent server-side loops share one SQLite database:

```
┌────────────────────────────────────────────────────────────────────┐
│ backend (Node 22, Express)                                         │
│                                                                    │
│  scheduler.js ──────────────┐          ingestScheduler.js (NEW)    │
│  (display, */10 min)        │          (ingestion, */60 min)       │
│         │                   │                   │                  │
│         ▼                   │                   ▼                  │
│  agent/displayAgent.js      │          ingest/pipeline.js (NEW)    │
│    ├─ contextAgent  ────────┼──reads──▶  ├─ gmail/fetch            │
│    ├─ renderAgent           │            ├─ prefilter              │
│    └─ validator (NEW)       │            ├─ attachments (pdf/pw)   │
│         │                   │            ├─ extractAgent (NEW role)│
│         ▼                   │            └─ store writers          │
│  display_cache              │                   │                  │
│                             ▼                   ▼                  │
│                    store/db.js  +  store/vault.js (NEW)            │
└────────────────────────────────────────────────────────────────────┘
```

Rules:

- **A1.** The display loop never calls Gmail/Calendar APIs. It reads only local stores. An empty/partial store is a valid state.
- **A2.** The ingestion loop never touches `display_cache`. The two loops share no locks beyond SQLite's own (WAL mode, already enabled).
- **A3.** Both loops keep the existing single-flight guard pattern (`isRunning` flag) — a slow ingestion run skips the next tick rather than overlapping.
- **A4.** All Google API calls go through one client module with token refresh; nothing else imports `googleapis`.

## 2. Encryption design (vault)

- **Algorithm:** AES-256-GCM via `node:crypto`. Per-value random 12-byte IV; stored format `v1:<iv b64>:<authTag b64>:<ciphertext b64>` in a TEXT column.
- **Key:** `VAULT_KEY` in `.env` — 32 bytes, base64. Generated by a setup script (`npm run gen-vault-key`) that appends to `.env` if absent. Backend refuses to start ingestion features (and logs a clear warning) if vault tables are non-empty but `VAULT_KEY` is missing/wrong (detected via a `vault_check` sentinel row encrypted at first write).
- **API surface:** `store/vault.js` exposes `putSecret(table, key, value)` / `getSecret(...)` only; no route or agent ever receives ciphertext or the key. Routes return masked values (`"98•••••210"`).
- **Never:** log decrypted values; include them in LLM prompts except the minimum fields needed to compute a document password (F1.3); send them to the frontend.

## 3. Google integration

### 3.1 OAuth (shared by Gmail + Calendar)

- One Google Cloud OAuth client (`GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET` in `.env`), scopes: `gmail.readonly`, `calendar.readonly`.
- Flow (device is a kiosk; browser available via admin panel):
  1. `GET /api/connections/google/start?label=personal` → returns Google consent URL (loopback redirect to backend).
  2. `GET /api/connections/google/callback` → exchanges code, encrypts refresh token into `oauth_tokens`, creates/updates `mail_accounts` row.
- Access tokens cached in memory only; refreshed on 401. Three consecutive refresh failures → account `status = 'error'`, surfaced in Connections tab and as an ingestion-stale warning card eligibility (PRD risk table).
- Disconnect: `DELETE /api/connections/:accountId` → revoke token with Google, delete `oauth_tokens` rows, keep `processed_emails` (audit) unless `?purge=true`.

### 3.2 Gmail fetch strategy

- Allowlist compiles to a Gmail search query per account: `(from:a@x.com OR from:b@y.com OR label:desk-bot) newer_than:7d`, plus `-in:spam -in:trash` always.
- Incremental sync via `historyId` per account (stored on `mail_accounts`); full query fallback when historyId expires.
- Fetch caps per run: ≤ 50 messages/account, attachment ≤ 10 MB, body truncated to 20k chars before LLM. Exceeding items are skipped+logged (`reason: 'cap'`).

### 3.3 Calendar sync

- Pull events for `now … +7d` per connected account on the ingestion cron; upsert into `events` with `source:'gcal'`, `sourceRef = eventId`. Deletions in Google remove the local row (reconcile by listing current window). Calendar wins over email-extracted duplicates (match on title+date±1h → drop the email fact, keep gcal).

## 4. Ingestion pipeline (`ingest/pipeline.js`)

Per account, per run:

```
fetch (3.2) → for each message not in processed_emails:
  prefilter →  discard | transactional | newsletter
  transactional:
    attachments? → attachments.js (text or password chain 4.2) 
    extractAgent → facts[] → store writers (dedup-safe upserts)
  newsletter:
    strip html → cap 8k chars → extractAgent (digest mode) → digest_items
  record processed_emails row (outcome, reason, factRefs)
```

### 4.1 Prefilter rules (no LLM)

- Sender not in allowlist → **discard** (should not occur; defense in depth).
- Allowlist type `newsletter` → **newsletter path**.
- `List-Unsubscribe` header present AND sender type `transactional` → **discard**, `reason:'marketing'`.
- `In-Reply-To`/`References` present (thread) → **discard**, `reason:'thread'`.
- Else → **transactional path**.

### 4.2 Password resolution chain (`ingest/passwords.js`)

1. **Email-stated formula:** extractAgent's first pass returns `passwordHint` (verbatim sentence describing the password) when present. `passwords.js` maps hint → formula over vault fields (PAN, DOB in `DDMMYYYY`/`DDMM`, mobile last-4, account last-N; case variants). Try up to 8 candidate strings.
2. **Known-sender formula table:** static map `senderPattern → formula` for major Indian brokers/banks, same vault-field composition.
3. **Stored per-sender password** (`document_passwords`, decrypted).
4. **Queue for user:** insert into password prompt queue (`processed_emails.outcome='skipped', reason='locked'`); admin panel Activity tab lets the user enter it → stored encrypted → message re-queued for the next run.
- Successful candidate is persisted to `document_passwords` (encrypted) with the sender pattern.
- Hard cap: never more than the candidates above; no incremental guessing.

### 4.3 Store writers & dedup

- Every write is an upsert keyed on `(sourceEmailId, factType, naturalKey)` — e.g., a bill's natural key is `(vendor, dueDate, amount)`. Re-processing a message (manual re-run) can never duplicate.
- All writes tagged `source:'email'`, `sourceEmailId`.
- Review queue = facts are written live (auto-accept per PRD §13.1) with `processed_emails.factRefs` populated; `DELETE /api/ingest/facts/:ref` removes a wrong fact and marks the email `outcome:'skipped', reason:'user-rejected'` so it is not re-extracted.

## 5. Agent & tool specifications

### 5.1 New model role: `extract`

`modelProvider.js` gains `"extract"` alongside `context`/`render`; env overrides `EXTRACT_LLM_PROVIDER/MODEL/API_KEY/BASE_URL`, falling back to globals. Intended for a cheap model.

### 5.2 extractAgent contract

Single `generateText` call (no tool loop), JSON output, two modes:

```jsonc
// transactional mode — input: {sender, subject, bodyText, attachmentText?}
{
  "intent": "transaction|bill|event|task|delivery|statement|other",
  "confidence": 0.0,
  "passwordHint": "verbatim sentence or null",
  "facts": [
    { "type": "bill",  "vendor": "", "amount": 0, "currency": "INR", "dueDate": "YYYY-MM-DD" },
    { "type": "event", "title": "", "date": "YYYY-MM-DD", "time": "HH:MM|null" },
    { "type": "task",  "title": "", "due": "YYYY-MM-DD|null", "priority": "medium" },
    { "type": "trade", "symbol": "", "side": "buy|sell", "quantity": 0, "price": 0 }
  ]
}
// digest mode — input: {sender, strippedText}
{ "items": ["headline 1", "headline 2"] }   // 2–4 items
```

- Prompt rule: *omit a fact rather than guess a field* (PRD risk mitigation). `confidence < 0.5` → treat as `other`, skip.
- Malformed JSON → one retry with a "fix your JSON" turn → else skip+log.

### 5.3 Context agent changes

- New tools: `get_bills()` (due/unknown bills, next 14 days), `get_digest()` (unexpired digest items).
- Priority ladder insert: `BILL — bills due within 3 days → MEDIUM` (between TASKS and PORTFOLIO); `inbox_digest` joins the AMBIENT/GENERAL band.
- `select_content` enum extends with `bill` and `inbox_digest`.

### 5.4 Render agent changes (primitives)

- `primitives/index.js` loads snippet files and emits one prompt block: name, one-line purpose, parameter slots, and the snippet. Injected into the render system prompt.
- Prompt directives: compose any subset; overriding styles and writing custom HTML is allowed; vary layout/arrangement/accent each cycle; avoid the layouts described by the last 5 `layoutFingerprint`s (provided as text).
- After generation, backend computes `layoutFingerprint = sha1(sorted primitive names used + gridSignature)` (primitive use detected via `data-prim="name"` attributes the snippets carry) and stores it on `history`.

### 5.5 Validator (`agent/validator.js`)

Checks (no LLM, <50 ms): parseable HTML (`node-html-parser`); single self-contained document; external refs limited to the Chart.js CDN allowlist; no `<form>`, no inline event handlers (`on*=`), `<script>` only for chart init (heuristic: scripts must reference `Chart(`); body dimensions match settings (style contains screen width/height or 100vw/vh). 
Fail → one render retry with the failure reasons appended → fail again → keep previous display, log, `display_cache` untouched.

## 6. API contracts (new routes)

All JSON; errors as `{ "error": { "code": "string", "message": "string" } }` with proper HTTP status.

| Method & path | Purpose | Notes |
|---|---|---|

| `GET  /api/connections/google/start?label=` | Begin OAuth | returns `{url}` |
| `GET  /api/connections/google/callback` | OAuth redirect target | HTML "connected" page |
| `DELETE /api/connections/:id?purge=` | Disconnect/revoke | |
| `GET/PUT /api/connections/:id/allowlist` | Manage sender/label entries (with `type`) | |
| `GET  /api/vault` | Masked identity fields | |
| `PUT  /api/vault` | Upsert identity fields | plaintext in over localhost; encrypted at rest |
| `DELETE /api/vault` | Wipe vault + document_passwords + tokens | confirm token required |
| `GET  /api/ingest/activity?limit=` | Recent processed_emails w/ outcome+reason | |
| `POST /api/ingest/run` | Manual ingestion trigger | 409 if running |

## 8. Error handling & resilience

| Failure | Behavior |
|---|---|
| LLM 429 (any role) | existing `withRateLimitRetry` (15/30/60 s, 4 attempts) reused for extract role |
| Gmail/GCal API error | skip account this run, mark last error on `mail_accounts`, continue others |
| Token refresh fails ×3 | account `status:'error'`; Connections tab badge; display may show a small "reconnect Gmail" ambient card after 24 h stale |
| PDF unparseable / image-only | skip, `reason:'unreadable'` (OCR is Phase 3) |
| Password unresolved | skip, `reason:'locked'`, prompt queue (4.2) |
| Extract JSON invalid ×2 | skip, `reason:'extract-failed'` |
| Validator fails ×2 | keep previous display (5.5) |
| NewsAPI 4xx/429/timeout | transparent RSS fallback, same article shape |
| DB busy/locked | SQLite WAL + busy_timeout 5000 ms |

Logging: every pipeline decision logs one line `[ingest] <msgId> <outcome> <reason>`; no email bodies, no secrets in logs.

## 9. Database migrations

`db.js` gains a `schema_version` pragma-style table and idempotent, ordered migrations:

1. `001` — add `source`, `sourceRef`/`sourceEmailId` columns to `tasks`, `events`, `portfolio` (DEFAULT `'manual'`); add `layoutFingerprint` to `history`.
2. `002` — create `bills`, `mail_accounts`, `allowlist_entries`, `oauth_tokens`, `processed_emails`, `digest_items`, `identity_vault`, `document_passwords`.
3. `003` — seed nothing; vault stays empty until user enters data.

Rules: additive only (SQLite `ALTER TABLE ADD COLUMN`); never destructive; existing Phase 1.5 DBs upgrade in place on first boot.

## 10. Testing strategy

- **Unit (node:test):** vault encrypt/decrypt round-trip + tamper detection; password formula computation (fixture hints → expected candidates); prefilter rules; store-writer dedup (same message twice ⇒ one fact); validator accept/reject fixtures; RSS normalization.
- **Integration (mocked HTTP):** pipeline end-to-end with fixture Gmail payloads (plain email → task; contract-note PDF with password → trade; newsletter → digest items; look-alike sender → never fetched); migration test upgrading a copied Phase 1.5 DB.
- **Golden fixtures:** a `test/fixtures/emails/` corpus (sanitized) is the regression suite for extraction prompts — any prompt change must keep fixture pass rate.
- **Manual/E2E:** one scripted smoke: connect account (mock), run ingestion, trigger display cycle, assert `bill` can be selected and rendered HTML passes the validator.
- Non-goal: no automated evaluation of subjective UI beauty; variety is checked via fingerprint uniqueness across a 10-cycle loop test with a stubbed model.

## 12. Non-functional acceptance (traceable to PRD §9)

- Display cycle p95 ≤ 90 s — measured via existing cycle logs + timestamps.
- Ingestion run (≤ 50 msgs) completes within one interval — timed in pipeline log.
- Render tokens −40% — `usage` from the AI SDK recorded per cycle before/after M5.
- Zero plaintext secrets — grep-able invariant: `identity_vault.valueEnc`, `document_passwords.passwordEnc`, `oauth_tokens.refreshTokenEnc` are the only secret columns and all start with `v1:`.
